# HyperWeave

A headless visual design system for AI agents. One API call composes self-contained SVG -- from status badges to telemetry strips to full branded component kits. Zero external dependencies. Living CSS state machines. Renders anywhere an `<img>` tag works.

## The Composition Formula

Every artifact is the output of a single formula:

```
ARTIFACT = FRAME x PROFILE x GENOME x SLOTS x MOTION x ENVIRONMENT
```

- **Frame**: The structural type (badge, strip, icon, divider, marquee-horizontal, receipt, rhythm-strip, stats, chart, matrix)
- **Profile**: Structural skeleton (corner radii, accent widths, typography scale, glyph backing)
- **Genome**: Aesthetic DNA (colors, shadows, glow, rhythm, compatible motions)
- **Slots**: Content filling frame zones (labels, values, metrics, glyphs)
- **Motion**: Animation primitive (static output or SVG-native SMIL border overlays)
- **Environment**: Target platform constraints (GitHub README, Notion, email)

## Why HyperWeave?

Agent-generated visuals are disposable -- no state, no brand, no memory. Every generation starts from zero. HyperWeave solves this structurally:

- **Semantic SVGs** with embedded CSS state machines -- artifacts respond to their own state
- **Genome system** -- portable, machine-readable aesthetic specs any agent can consume
- **Self-contained** -- zero external references, no runtime JS, no CDN dependencies
- **Three interfaces** -- CLI, HTTP API, and MCP tools for any integration pattern
- **Telemetry** -- parse supported agent sessions into structured visual receipts

## Architecture at a Glance

```
ComposeSpec -> engine.py -> assembler.py (CSS) -> lanes.py (validate) -> templates.py (Jinja2) -> SVG
```

Python builds context dicts. Jinja2 builds SVG. YAML defines config. Three layers, no mixing.
